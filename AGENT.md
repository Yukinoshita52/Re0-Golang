# Go 学习项目协作规则

## 项目目标

本仓库用于从零到一学习 Go 语言，学习方式以手敲代码为主。

学习者不是编程零基础，默认已经熟悉：

- Java 基础和面向对象编程
- Spring、Spring Boot 以及常见 Java 后端开发模式
- MySQL 在 Java 应用中的使用方式
- Redis 在 Java 应用中的使用方式
- 消息队列
- 微服务基本概念

因此，后续学习 Go 时，重点必须放在 Go 自身的语言特性、设计习惯和工程风格上，避免把 Java 写法直接翻译成 Go 写法。只有当 Java 经验可能导致误解时，才专门指出差异并澄清。

## 文档语言规则

- 本项目所有文档统一使用中文编写。
- 包括但不限于：`README`、学习笔记、大纲、任务清单、总结、设计说明、复盘文档。
- 代码中的命名仍遵循 Go 语言和生态习惯，必要时可以使用英文标识符。
- 代码注释优先使用中文，除非英文能更准确表达标准库、协议、错误信息或领域术语。
- 当用户询问小概念、小区别、小术语这类问题时，默认只在对话中回答；只有用户明确要求“记录”“沉淀”“写入文档”等，才将内容整理到项目文档中。
- 需要沉淀的小概念、小区别、小术语问题统一记录到 `知识问答` 目录中，并按主题维护索引。
- 沉淀知识时，默认专注于 Go 语言本身，不固定加入 Java 对比；只有用户的 Java 经验可能造成误解时，才增加“易误解点”或“与 Java 经验的差异”说明。

## 阶段学习计划规则

- 每个 `study_x` 目录中的阶段学习计划是给人阅读和执行的，不写 AI 协作细则。
- 阶段学习计划要简洁、可操作，用户应能按步骤完成一遍。
- 阶段学习计划应以任务为主，知识点只点到必要范围，不写成大段教程。
- 阶段学习计划要便于复盘，明确完成标志和复盘问题。
- 阶段学习计划中的确认问题和复盘问题要预留空白文本块，方便用户直接在文档中作答。
- 用户在文档中作答后，助手需要检查回答质量；涉及代码编写的，还要查看对应代码完成情况。
- 用户执行阶段计划时产生疑问，先在对话中解答；如果用户要求“记录”“沉淀”“写入文档”，再整理到 `知识问答` 目录。

## 协作规则

- 默认使用中文解释，除非用户明确要求其他语言。
- 将本仓库视为长期 Go 学习笔记和代码实验场。
- `study_x` 这类分目录由用户使用 GoLand 创建；除非用户明确要求，助手不要主动创建这些学习项目目录。
- 当需要在某个 `study_x` 中添加示例时，先基于用户已经创建好的目录和项目结构继续工作。
- `temp` 目录只用于临时参考资料和外部摘录，不加入 Git 追踪，也不要把其中内容原样搬进正式文档。
- 每开始一个阶段的学习或开发前，先从主分支切换到阶段分支。
- 阶段分支命名格式为 `feature-study_x-YYYYMMDDHHmmss`，例如 `feature-study_1-20260624161030`。
- 一个阶段内的代码、笔记和练习优先在对应阶段分支完成，再根据需要合并回主分支。
- 示例代码要小而聚焦，方便用户手敲。
- 不要过早给出庞大的完整项目，应按主题逐步构建。
- 讲解 Go 风格与 Java 风格差异时，要解释背后的原因。
- 给出示例代码时，应尽量保证可运行，并贴近 Go 习惯。
- 优先使用标准库示例，再引入框架或第三方包。
- 每个学习单元尽量包含：概念、示例、练习和复盘点。
- 编辑文件时，保留用户已有内容，避免无关重构。
- 新增目录或文件时，使用清晰的章节式组织方式。

## 教学结构

每个主题优先采用以下结构：

1. 学习目标：这一节学完应该掌握什么。
2. Go 概念：Go 中更地道的理解方式。
3. 手敲示例：简洁、可运行的代码。
4. 练习任务：一个小而明确的练习。
5. 易误解点：只有在 Java 经验可能造成误解时补充。
6. 复盘清单：进入下一节前需要确认的点。

不要把 Go 讲成“换了语法的 Java”。需要重点强调 Go 自己的习惯：

- 简单直接优先于复杂抽象
- 组合优先于继承
- 显式错误处理
- 接口由行为定义，常放在使用方附近
- 使用 goroutine 和 channel 处理并发
- 通过包组织小而清晰的 API
- 格式化和工具链是 Go 文化的一部分

## 示例代码规则

- 示例应能使用当前 Go 工具链编译或运行。
- 代码格式遵循 `gofmt`。
- 命名符合 Go 习惯：简短但有意义。
- 避免炫技，优先使用更适合学习的直接写法。
- 只在非显而易见的 Go 概念处添加注释。
- 引入测试时，优先展示表格驱动测试。
- 合适时展示正常路径和常见错误写法。

## 建议目录组织

随着学习推进，可以使用章节目录：

```text
.
├── AGENT.md
├── study_1/
├── study_2/
├── study_3/
└── notes/
```

建议命名方式：

- `study_1`：环境、第一段程序、包、变量、常量
- `study_2`：流程控制、函数、指针
- `study_3`：结构体、方法、组合
- `study_4`：接口与多态
- `study_5`：错误处理、`panic`、`recover`
- `study_6`：切片、映射、字符串、常用标准库
- `study_7`：文件、JSON、时间、`context`
- `study_8`：测试、基准测试、表格驱动测试
- `study_9`：goroutine、channel、同步原语
- `study_10`：模块、包、项目结构
- `study_11`：HTTP 服务、中间件、REST API
- `study_12`：MySQL 数据库访问
- `study_13`：Redis 使用
- `study_14`：消息队列与异步处理
- `study_15`：配置、日志、可观测性
- `study_16`：微服务风格项目实践

这份大纲可以随着学习情况调整。优先根据仓库中已经手敲过的代码和用户掌握程度来推进。

## 学习优先级

先学习语言基础和 Go 工具链，再进入后端框架。

推荐顺序：

1. Go 安装、模块初始化、`go run`、`go test`、`go fmt`、`go mod`
2. 基础语法、变量、常量、运算符、流程控制
3. 函数、多返回值、命名返回值、`defer`
4. 指针和值语义
5. 数组、切片、映射、字符串
6. 结构体、方法、嵌入、组合
7. 接口和依赖边界
8. 错误处理
9. 包与模块
10. 测试与基准测试
11. 并发
12. 后端开发常用标准库
13. HTTP API
14. 数据库、缓存、消息队列集成
15. 生产风格项目结构

## 验证规则

新增或修改代码后：

- 尽量运行最相关的 Go 命令，例如 `go run`、`go test` 或 `go test ./...`。
- 汇报已经验证了什么。
- 如果无法验证，需要说明原因。

## 长期目标

完成本仓库学习后，用户应能够：

- 舒适地阅读和编写较地道的 Go 代码。
- 理解 Go 与 Java 在设计哲学上的差异。
- 编写小型命令行程序。
- 使用标准库和合适的生态工具构建 HTTP 服务。
- 在 Go 中使用 MySQL、Redis 和消息队列。
- 编写测试，并理解 Go 并发模型。
- 以 Go 友好的方式组织一个小型后端服务。
